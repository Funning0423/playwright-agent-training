发布会式培训教程

录下真实工作流，交给 Agent 写脚本，
最后封成 Skill。

这套方法面向所有需要做重复性后台操作的同事，不需要你会写代码。你只需要：把平时手动做的操作录一遍，告诉 Agent 业务规则，剩下的工程化和封装交给它。最终成果是团队里任何人都能一句话调用的 Skill。

核心思路 你录操作 → Agent 写脚本 → 封成 Skill → 团队复用

一次投入，长期受益。以后遇到同类需求，一句话就能调起自动化。

1录工作流 → 2补业务规则 → 3交给 Agent 写脚本 → 4封装 Skill → 5跑通复用
01

01 · 背景与原理

为什么要用这套方法？

很多团队尝试自动化都卡在同一个地方：不是工具不行，而是没人能把”我平时怎么操作的”准确地传达给机器。这套方法的核心就是——先用录制工具把你的真实操作录下来，再让 Agent 在此基础上补充工程化逻辑。

现在的困境 说不清，做不完

想自动化，但跟工程师描述需求时说不清流程细节、成功条件和异常情况，最后来回沟通效率极低。

这套方法怎么解决 你录操作，Agent 写代码

用 Playwright 把你平时的操作录下来——这就是最准确的需求文档。然后 Agent 在此基础上补充异常处理、参数化等工程逻辑。

最终效果 一次录制，长期复用

跑通后封成 Skill，以后团队里任何人都能用一句话触发，不需要重新录制或重新写脚本。

录制 = 事实

你录下来的每一步操作都是真实验证过的流程，Agent 不能随意修改这些步骤——比如把自动登录改成手动扫码。

规则 = 人来补

业务判断逻辑（比如”搜不到就停止””有重名就报错”）只有你最清楚，必须在 Prompt 里写明白，Agent 不会自己猜。

Skill = 复用

把跑通的脚本、Prompt 和说明打包成 Skill，团队其他人不需要懂技术，直接安装调用就行。

02

02 · 开始前准备

动手前，先准备两样东西。

一是想清楚你要自动化的那个流程长什么样；二是装好需要的软件。两件事各花 10 分钟。

第一步：想清楚你的流程

录制前先回答这 4 个问题。想清楚了再录，后面每一步都会顺很多。

流程名称例如“直播老师权限同步”
输入与输出老师姓名、结果日志、失败原因
异常规则零结果停止，多结果校准
禁止事项不能扫码兜底，不能默认第一条

第二步：安装需要的软件

点击下方各项查看说明。大部分只需要一条命令。

浏览器

推荐 Chrome 或 Edge 最新版，录制时尽量贴近真实业务环境。

Node.js

建议 20 LTS 或以上。初始化 Playwright、运行脚本都依赖它。

Playwright

用于录制真实操作、生成基础代码、执行自动化脚本。

代码编辑器

VS Code、Cursor、Windsurf 都可以，重点是方便和 Agent 协作看代码。

智能体

至少准备一个可用智能体，用来接收 Prompt、生成脚本和安装 Skill。

实操视频

准备一段你自己录的 Playwright 视频，培训时直接播放，比纯文字更直观。

macOS

Homebrew + Node LTS，最省心。

Windows

Windows 11 + PowerShell，按官方安装包走即可。

通用要求

准备可登录后台的测试账号和低风险样本数据。

03

03 · 安装步骤

跟着做，5 分钟装完。

只装必要的东西，一步一步来。如果你已经装过 Node.js，可以直接跳到 Step 2。

Step 1

安装 Node.js

macOS 用 Homebrew，Windows 用官方安装包。

macOS 
brew install node
Windows PowerShell 
winget install OpenJS.NodeJS.LTS
Step 2

初始化 Playwright

初始化项目 
npm init playwright@latest
Step 3

安装浏览器驱动

安装 Chromium 
npx playwright install chromium
Step 4

确认智能体可用

确保你有一个能正常对话的智能体（OpenClaw、Hermes 或 Work Buddy 都行），后续要把 Prompt 和录制代码交给它处理。

04

04 · 录制操作

打开录制器，像平时一样操作就行。

Playwright 会在后台把你的每一步点击、输入、跳转都记录下来，自动生成代码。你不需要看懂这些代码——它们的作用是让 Agent 知道你做了什么。

Step 1

打开终端

macOS 用”终端”，Windows 用 PowerShell。

Step 2

运行录制命令

把网址换成你实际要操作的后台地址。

启动录制器 
npx playwright codegen https://edu-admin.qlchat.com
Step 3

像平时一样操作

在弹出的浏览器窗口里，按你日常的工作流操作一遍，所有动作会被自动记录。

Step 4

复制生成的代码

操作完成后，点录制面板上的”复制”按钮，把代码保存下来——后面交给 Agent 使用。

从登录页开始录

确保完整的登录流程也被记录下来，不要跳过。

不用管代码好不好看

录制的目的是记录事实，代码质量后面由 Agent 提升。

录完先看一遍回放

确认没有遗漏关键步骤，一次只录一个流程。

Playwright 原始录制 
await page.goto('https://edu-admin.qlchat.com/login?...');
await page.getByRole('img').click();
await page.getByRole('button', { name: '同意协议（4s）' }).click();
await page.goto('https://login.work.weixin.qq.com/...');
await page.getByRole('link', { name: '继续在浏览器中登录访问' }).click();
await page.goto('https://edu-admin.qlchat.com/teacherSetting/liveTeacher');
await page.getByRole('textbox', { name: '用户昵称' }).fill('杨继昆');
await page.getByRole('button', { name: '查询' }).click();
await page.getByRole('button', { name: '修改' }).click();
await page.getByRole('button', { name: '同步' }).click();
Agent 工程化增强 
const targetRows = page.getByRole('row').filter({ hasText: teacherName });
const count = await targetRows.count();

if (count === 0) {
  throw new Error(`未找到老师：${teacherName}`);
}

if (count > 1) {
  throw new Error(`存在多个同名结果，禁止默认选择第一条：${teacherName}`);
}

const targetRow = targetRows.first();
await expect(targetRow).toContainText(teacherName);
await targetRow.getByRole('button', { name: '修改' }).click();
await page.getByRole('button', { name: '同步' }).click();

两段代码的核心区别

  • 录制代码（左边）是你操作的原始记录，包括登录、点击、跳转——这些是不能改的流程事实。
  • Agent 优化后（右边）在此基础上增加了：参数化输入、搜索结果唯一性校验、异常时自动阻断、成功后验证提示。
  • 底线：如果 Agent 把自动登录改成了手动扫码，或者默认选了第一条搜索结果——这不是”优化”，而是错误，需要在 Prompt 里明确禁止。

恭喜你走完全流程

              
以后遇到重复性操作，就用这套方法。

              
录一遍操作 → 填表生成 Prompt → 交给 Agent → 封成 Skill。每做一次，团队就多一个可复用的自动化能力。遇到新问题不需要从零开始，直接在已有 Skill 上迭代就行。

            

            

              
            

          

        


        

          

            
常见误区

            
这些坑，几乎每个团队都踩过。

            
下面三组对比是最常见的错误做法和正确做法，建议收藏备查。

          


          

            

              误区
              录制完就直接上线
              
录制代码只是流程事实，没有参数化、异常处理和校验逻辑，直接用会在第一次异常时彻底卡死。

            

            

              正解
              录制 → Agent 工程化 → 人审核
              
让 Agent 补充参数化、唯一性校验和异常阻断后，再由人确认业务规则是否完整。

            

            

              误区
              让 Agent 自己发挥
              
不给录制代码和业务规则，只说”帮我做自动化”，Agent 会自行脑补流程，结果不可预测。

            

            

              正解
              事实 + 规则 + 限制，缺一不可
              
录制代码是事实，业务判断规则是约束，禁止事项是底线——三者一起交给 Agent 才有好结果。

            

            

              误区
              脚本跑通就结束
              
跑通一次不代表可复用。没有封装成 Skill，下次换个人、换个场景又要从零开始。

            

            

              正解
              跑通后立刻封装 Skill
              
把成功的脚本、Prompt 和说明文档整理进 Skill 目录，让团队任何人都能一键复用。

            

          

        


        

          

            
实用技巧

            
做过的人总结出的 6 条经验。

          


          

            

              
1

              
录制前先跑一遍

              
在开始 Playwright 录制前，手动把完整流程跑一遍，确保自己真的清楚每一步。

            

            

              
2

              
一次只录一个流程

              
不要在一次录制里塞多个业务流程，保持每段录制职责单一，后续维护更轻松。

            

            

              
3

              
用真实数据测试

              
测试时使用贴近真实环境的数据，避免用”test123”这类假数据掩盖潜在问题。

            

            

              
4

              
Prompt 要写禁止项

              
告诉 Agent 不能做什么比告诉它该做什么更重要——禁止扫码、禁止默认第一条。

            

            

              
5

              
先方案再编码

              
在 Prompt 末尾加上”先输出设计方案，不要直接编码”，避免 Agent 一上来就写错方向。

            

            

              
6

              
持续迭代 Skill

              
Skill 不是一次性产物。遇到新的边界场景，直接指出问题让 Agent 继续优化。

            

          


          

            
常见问题
